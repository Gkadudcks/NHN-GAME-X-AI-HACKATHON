# 단서 시스템 구조

## 표준 데이터 계약

모든 런타임·저장·스토리 단서는 아래 다섯 필드만 가진 레코드로 취급한다.

```js
{
  id: "d2_verified_baseline",
  day: 2,
  theme: "검증 기록",
  title: "DAY 2 검증 기준",
  detail: "최근 7일 신규 설치 12,480명..."
}
```

- `id`: 문구가 바뀌어도 유지하는 의미 기반 불변 ID다.
- `day`: 1~5 정수이며 화면은 이 값으로 날짜를 분류한다.
- `theme`: 마인드맵 가지 이름이다.
- `title`: 노드와 일일 요약에 표시하는 짧은 제목이다.
- `detail`: 상세 패널에 표시하는 독립적인 설명이다.

정식 단서는 `NAN_GAME_TITLE/js/clue-records.js`의 중앙 카탈로그에 한 번만 정의한다. DAY별 스토리는 `ClueRecords.get(id)`로 레코드를 받아 사용하며, 문자열·정규식·배열 위치로 메타데이터를 추론하지 않는다.

## 저장과 마이그레이션

- 공유 진행 스키마는 v2이며 단서를 레코드 배열로 저장한다.
- canonical v1 문자열 단서는 로드 시 v2 레코드로 변환하고 같은 저장 키에 승격한다.
- DAY 1 로컬 저장은 v4로 기록하며 기존 v3 저장도 읽어 레코드로 변환한다.
- 알려진 구문은 중앙 alias를 통해 정식 ID로 합친다.
- 알 수 없는 과거 문자열은 `legacy_d{day}_{hash}` ID로 보존하며 삭제하지 않는다.
- DAY 2 신규 단서는 스냅샷 배열 길이가 아니라 스냅샷 ID 집합과의 차이로 계산한다.

`clue-records.js`는 `progress-store.js`, `day2-story.js`, `clue-mindmap.js`, 각 DAY 엔진보다 먼저 로드해야 한다. 캐시 쿼리 버전도 이 의존 순서를 변경한 HTML과 함께 올린다.

## 새 단서 추가 방법

1. 중앙 카탈로그에 고유하고 의미 기반인 ID로 레코드를 추가한다.
2. 스토리 장면에는 문자열 대신 `ClueRecords.get(id)` 결과를 연결한다.
3. 같은 제목의 다른 단서는 다른 ID로 정의하고, 같은 사건의 문구 수정은 기존 ID를 유지한다.
4. 구버전 저장 문구를 흡수해야 하면 해당 레코드의 `legacy` 또는 `legacyPrefix`만 추가한다.
5. `clue-records.test.js`, `progress-store.test.js`, 관련 DAY 스토리 테스트와 실제 브라우저 검증을 실행한다.

## 이번 구조 변경

- DAY 1·2 단서 생산자를 중앙 카탈로그 레코드 참조로 전환했다.
- 공유 진행 schema v1과 DAY 1 save v3에 대한 무손실 마이그레이션을 추가했다.
- 중복 제거, 날짜 분류, DAY 2 일일 차집합을 모두 안정적인 ID와 명시 필드 기준으로 바꿨다.
- 마인드맵에서 문자열 기반 날짜·테마·제목·상세 추론을 제거했다.
- 반복 렌더 시 이전 `ResizeObserver`를 해제하고 데이터·저장·뷰 회귀 테스트를 추가했다.
